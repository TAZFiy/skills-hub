# Lessons

- 2026-03-23: 如果项目要求先查看 lessons，而文件不存在，应先创建基础文件并在后续任务结束时补充本次经验，避免流程断点。
- 2026-03-23: 对前端重设计类任务，优先复用现有数据与行为层，只替换视觉系统和信息架构；这样验证面更小，能更快确认问题是否来自样式层还是业务层。
- 2026-03-23: 当用户明确否定“工业风 / 高端产品风 / 实验风”等风格选项时，应立刻回到工具本质，先确认工作模式和信息密度，再决定视觉表达；资源管理工具优先考虑目录、列表、路径和 inspector，而不是 hero 或情绪化视觉。
- 2026-03-23: 对列表/目录管理工具，默认不要保留常驻说明文案或固定详情栏；优先让详情按需弹出，界面正文只保留导航、筛选、列表和操作。
- 2026-03-23: 当页面需要承担“编辑 Markdown 规则”职责时，布局应优先服务编辑器本身；把筛选、元信息和创建参数压缩到侧栏或工具条，主区域留给编辑 / 预览 / 分栏工作流。
- 2026-03-23: 如果用户说“我要的是网页的 md 编辑器”，不要停留在“文件管理器里嵌一个 textarea”的思路；默认按网页 IDE 理解，优先确认文件导航、默认编辑态和最小工具集，再决定是否保留管理表单。
- 2026-05-10: 当用户把“只需要支持编辑现有全局规则文件”说得很明确时，应同步收紧 UI 和后端 allowlist；只删前端入口不够，扫描与保存边界也要一起收口。
- 2026-05-10: 对资源列表类工具，高频操作如果落在详情抽屉里，哪怕功能完整也会显得笨重；应优先前移到行内，详情只保留补充信息。
- 2026-05-10: loading 反馈对管理工具非常关键；即使不做复杂动画，也应至少补齐 spinner、处理中文案和 disabled 态，减少用户对点击是否生效的疑虑。
